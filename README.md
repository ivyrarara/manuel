# 마늘 (Maneul)

100일간의 여정 파트너. 텔레그램 에이전트.

> 단군신화에서 곰은 100일간 동굴에서 마늘을 먹고 사람이 되었다.
> 마늘은 그 100일 동안 한 마디도 하지 않았다. 그냥 매웠을 뿐이다.

## 이게 뭔가

매일 밤 9시에 스스로 깨어나서, 최근 대화와 액션 상태를 읽고
**말을 걸지 말지를 스스로 판단합니다. 대부분의 날은 침묵합니다.**

그리고 자기 판단을 전부 기록합니다 — 무엇이 방아쇠였는지, 얼마나 확신했는지,
침묵했다면 뭐라고 말하려다 삼켰는지, 그리고 그 판단이 맞았는지.

**코드가 아니라 `checkins` 테이블이 이 프로젝트의 자산입니다.**

## 배포

→ **[배포하기.md](배포하기.md)** 를 보세요. 터미널 없이 웹에서만 합니다.

## 구조

```
agent/
  config.py    배경, 스케줄, 피드백 어휘
  db.py        SQLite 저장소  ← checkins 테이블이 핵심
  brain.py     프롬프트 조립 + Claude 호출
  checkin.py   말할지 침묵할지 판단  ← 이 프로젝트의 본체
  main.py      텔레그램 핸들러 + 스케줄
```

`checkin.py`의 `CHECKIN_SYSTEM`을 튜닝하는 게 100일 동안 할 일입니다. 나머지는 배관입니다.

## 침묵이 기본값

매일 깨어나지만 대부분의 날은 아무 일도 없어야 합니다. 매일 말을 걸면
알림은 배경소음이 되고, 마늘은 무시당하기 시작합니다.

코드 차원의 안전장치: **답이 없는데 2번 연속 말을 걸었으면 모델 판단과 무관하게
물러납니다** (`BACKOFF_THRESHOLD`). 프롬프트에 맡기기엔 중요한 규칙입니다.
판단이 깨지거나 파싱에 실패해도 침묵합니다.

## 타임존

`TIMEZONE`은 **지금 실제로 있는 곳**으로 설정하세요. 스케줄과 저장 시각 전부가 여기 맞춰집니다.
틀리면 밤 9시 체크인이 아침에 울립니다. 이사하면 이 값 하나만 바꾸면 됩니다.

시각은 Python이 이 타임존으로만 찍습니다. SQLite의 `datetime('now')`는 UTC라서
섞어 쓰면 '최근 7일' 계산이 시차만큼 어긋납니다.

## 페이스메이커

마늘은 심판이 아니라 페이스메이커입니다. 그러려면 목표가 필요한데,
**목표는 개수가 아니라 도달 깊이**입니다.

| 단 | |
|---|---|
| 1 | 학습 — 알게 됨 |
| 2 | 활용 — 실제로 써봄 |
| 3 | 확장 — 다른 도구나 AI로 옮겨봄 |
| 4 | 구현 — 만들어냄 |
| 5 | 개선 — 쓰면서 고침 |
| 6 | 기록 — 남에게 내놓음 |

한 주에 하나를 해도 6단까지 갔으면 좋은 주. 열 개를 배우기만 했으면 얕은 주.
육아휴직 중에 개수 할당량은 정직한 기준이 아닙니다.

4주 연속 3단 이하에서 맴돌면 `pace` 방아쇠가 당겨집니다.

**인플레이션 방지**: "했다"고 말한 것만 성취입니다. 계획, 의도, "알아봤다"는
전부 제외. 후하게 세면 차트가 거짓말이 되고 자기위안만 남습니다.

## 자동으로 쌓이는 성취

손으로 기록하는 건 2주면 안 합니다. 그래서 마늘이 알아서 봅니다.

| 출처 | 잡히는 것 | 단 |
|---|---|---|
| GitHub | 새 저장소 | 4단 구현 |
| GitHub | 커밋 (하루 = 1건) | 5단 개선 |
| 블로그 RSS | 새 글 발행 | 6단 기록 |
| 대화 | 마늘이 직접 감지 | 1~6단 |

**GitHub**: `GITHUB_USER`만 넣으면 됩니다. Events API 호출 한 번으로 **모든 공개
저장소**를 봐서, 100일 동안 새 앱을 만들어도 설정을 안 바꿔도 됩니다.
하루에 9번 커밋해도 성취는 1건입니다 — 커밋 수를 세면 숫자만 예뻐집니다.
그리고 커밋 수는 **성과가 아니라 마찰**입니다. 원하는 수정이 안 돼서 반복하는 거니까요.
1커밋은 알고 있었다는 뜻, 9커밋은 아홉 번 틀렸다는 뜻. 그래서 `(8번 만에)` 로 적습니다.
커밋 메시지가 전부 `Add files via upload` 같은 자동 문구면 그건 버리고
사람이 쓴 메시지를 골라냅니다.

## 블로그

`BLOG_RSS_URL`을 넣으면 발행한 글이 **자동으로 6단 성취**가 됩니다. 손 안 댑니다.

`/blog` 로 연결을 확인하세요. 첫 실행은 기존 글을 성취로 세지 않습니다 —
과거 글을 오늘 성취로 넣으면 페이스 차트가 1일차부터 거짓말이 됩니다.

같은 글을 고쳐서 재발행해도 성취는 1건입니다. 주소(guid)가 그대로니까요.
커밋과 같은 원리입니다 — 반복은 잘못된 걸 고치는 과정이지 새 성과가 아닙니다.

### 글 1개보다 중요한 것

**숫자는 세면 그만입니다. 놓친 통찰은 다음 글의 시작이 됩니다.**

글을 발행하면 마늘이 본문을 읽고 짚어줍니다:

```
「클로드 플랜을 구독했다」 읽었어요.

이 글은 GUI를 못 한다는 고백으로 시작하는데,
정작 본문 대부분은 브랜드를 감각으로 설계해온 이야기입니다.

놓치신 것 같은 지점:
💡 글이 스스로 증명하는 건 결핍이 아니라 자산인데,
   본인은 그걸 결핍의 언어로 쓰고 있어요.

다음 글이 될 수 있는 것:
🌱 에이전트의 침묵을 브랜드 보이스로 다루는 글
```

칭찬·요약·감상은 금지입니다. 글 안에 근거가 없으면 지어내지 않고, 없으면 침묵합니다.
RSS 본문이 얇으면 원문을 가지러 갑니다.

## 채점

마늘이 말을 걸면 메시지 아래 버튼 6개가 붙습니다. 탭 한 번, 타이핑 0.

| | |
|---|---|
| 🎯 맞다 | 정확했다 |
| 🌶️ 아픈데 맞다 | **가장 비싼 데이터** — 성공 |
| 🤷 뻔하다 | 새로울 게 없었다 |
| ⏰ 맞는데 지금 아님 | 타이밍 실패 |
| 🙄 강압적 | 맞았지만 방식이 틀림 — 실패 |
| ❌ 사실이 틀림 | 근거가 틀림 |

🌶️와 🙄의 차이가 이 프로젝트의 전부입니다. 둘 다 아픈데 하나는 성공이고 하나는 실패입니다.

**그리고 침묵도 채점됩니다.** 일요일마다 이번 주 삼킨 말들을 보여주고
"말했어야 했나?"를 묻습니다. 말한 것만 채점하면 데이터셋이 반쪽입니다.

## Raw 저장 우선

**판단 전에 먼저 기록. 원문은 절대 변형하지 않습니다.**

- `checkins.raw_input` — 그때 마늘이 본 것 전부 (상황 + 대화 이력)
- `blog_posts.content` — 글 원문
- `messages` — 대화는 그 자체가 원문
- `prompt_versions.template` — 그때의 규칙 전문

**왜**: 판단만 남기면 "v4가 v1보다 낫다"를 증명할 수 없습니다.
같은 입력에 두 버전을 돌려봐야 하는데 입력이 없으니까요.
원문이 있으면 나머지 개선은 전부 나중으로 미룰 수 있습니다 — 소급이 되니까요.
원문이 없으면 아무것도 소급되지 않습니다.

`/data` 로 원문 보존율을 확인하세요. 100%가 아니면 재현이 안 됩니다.

## 프롬프트 버전

판단 기준을 고칠 때마다 버전이 **자동으로** 바뀝니다 (규칙 + 학습된 요구사항의 해시).
사람이 버전을 매길 필요가 없습니다 — 지키지 않을 규율이니까요.

원본 전문이 `prompt_versions` 테이블에 보관되므로, 나중에 "그때 마늘은 어떤
규칙으로 판단했나"를 되짚을 수 있습니다. 해시만 있으면 해석이 불가능합니다.

`/data` 로 버전별 판단 횟수를 봅니다.

## 학습

대화 중에 "이렇게 해줘 / 이건 하지마" 라고 하면 자동으로 감지되어
`preferences`에 저장되고, **그 다음 모든 대화와 모든 체크인 판단에 들어갑니다.**
잘못 학습된 건 `/prefs` → `/forget <번호>`.
